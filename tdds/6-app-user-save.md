## 术语解释 {#terms}

以下为此文档会使用到的术语，在后续描述中，不再另行说明。

以下术语的目的是便于此文档中的后续说明，在 `TDD-6` 之外的场景下，这些术语可能含有其他含义，请注意区分。

- `存档系统`：轻量级的代码存储功能，设计上类似于 GitHub Gists。每个代码片段可包含一段或多段代码，适用于快速保存、复用或分享小型代码逻辑。是此文档中描述的所有实现方式的统称。
- `存档列表`：存档系统具体实现中，存档所在的列表界面或列表数据。在具体的实现方案中，此术语有不同的名称，在介绍具体方案时会进行额外说明。
- `存档条目`：存档列表中，单条存档记录称为存档条目。在具体的实现方案中，此术语有不同的名称，在介绍具体方案时会进行额外说明。
- `存档条目明细`：一个存档条目中，可包含一条或多条数据，每一条数据称为一个存档条目明细。在具体的实现方案中，此术语有不同的名称，在介绍具体方案时会进行额外说明。

以下术语可在 `TDD-6` 之外使用，在本地图软件中不容易产生歧义。

- `存档名称`：用户给存档起的易于记忆的名称，用于区分不同的存档记录。

## 地图存档 {#map-save}

为了便于本地图用户在不同的地图应用、不同设备之间共享数据，本地图使用存档系统实现用户存档。

### 存档数据模型 {#save-model}

``` typescript
type PrefData = T;

type MarkerIdData = number[];

type MarkerRefreshTimeData = Record<number, string>;
```

- `PrefData`：配置项类型，由于每个地图实现方案都会有软件自己的配置项，因此，此类型由具体实现自行决定。
- `MarkerIdData`：已标记点位的 ID 列表。
<!-- FIXME: 优化时间存储格式 -->
- `MarkerRefreshTimeData`：已标记点位中，需要刷新的点位的刷新时间映射。键名为点位 ID，键值为字符串类型的点位刷新时间。

### 存档数据 {#save-data}

对于地图数据存档，需要存储以下数据：

- 地图应用的配置，类型为 `PrefData`。
  - 请注意，由于此数据是地图软件自身的配置，需和其他地图实现方案的配置区分开，因此需要和其他开发者沟通，避免使用同样的存储名称导致覆盖。
  - 在此文档后续的描述中，这个场景下对存储名称的沟通过程，简称为“协商确认”。
- 已标记的点位 ID 列表，类型为 `MarkerIdData`。
- 已标记点位的刷新时间，类型为 `MarkerRefreshTimeData`。

此外，由于存档系统会使用第三方实现，所以需要注意：
- 第三方实现中，存档名称可能会被使用于存档条目名字中，因此需要根据第三方具体的规则，避免 **不同存档** 重名覆盖的情况发生。
- 第三方实现中，并不能保证所有数据都是用户存档数据。因此，在获取存档列表数据时，有必要对原始数据进行过滤，过滤规则会在具体实现方案中说明。

### 存档系统 {#save-sys}

存档系统可以使用不同的实现方案，以下是目前已有的方案。

#### Gitee：代码片段 {#save-sys-gitee}

Gitee 使用代码片段实现存档系统。

- 以下为此实现中，术语的具体名称：
  - `存档列表` → `代码片段列表`
  - `存档条目` → `代码片段条目`
  - `存档条目明细` → `代码片段条目文件`
- 技术对接：
  - Gitee 对接使用 [Gitee OAuth][gitee-oauth] 鉴权和 [Gitee API][gitee-api] 中的【代码片段】，实现存档系统的功能。
- 代码片段权限：私有
- 数据存储位置：
  - 地图应用的配置，保存到：
    - 代码片段条目：`Save_KYJG`。
    - 代码片段条目文件：协商确认。
  - 已标记的点位 ID 列表，保存到：
    - 代码片段条目：存档名称。
    - 代码片段条目文件：`Data_KYJG`。
  - 已标记点位的刷新时间，保存到：
    - 代码片段条目：存档名称。
    - 代码片段条目文件：`Time_KYJG`。
- 列表过滤规则：仅过滤代码片段条目中存在 `Data_KYJG` 的项。

[gitee-oauth]: https://gitee.com/api/v5/oauth_doc
[gitee-api]: https://gitee.com/api/v5/swagger

#### Q & A {#save-sys-qa}

- 为什么不自建存档服务？
  - 自建存档服务存在开发和维护成本。随着日益增长的用户量，团队没有足够的能力保证所有用户的使用体验，甚至没有足够的资源保证自建服务的可用性。在此情况下，第三方方案是最合适的选择。
